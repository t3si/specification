# 050.100-001 Core Definitions

## 050.100-001::010 Structural Primitives
The following types define the nature of T3S artifacts and dictate the validation logic applied by the system.

| type | definition |
| :--- | :--- |
| `t3s:graph` | The total logical representation of all system entities and their transversal relationships. |
| `t3s:root` | A top-level functional domain (000-900 series) acting as a primary entry point for a T3S block. |
| `t3s:branch` | An intermediate organizational node that groups related leaves or sub-branches. |
| `t3s:leaf` | A terminal node representing a specific standard, narrative, or data file. |
| `t3s:registry` | A specialized leaf containing an authoritative list of identities or keys. |
| `t3s:manifest` | The serialized, authoritative state of the `t3s:graph` used to map URNs to the physical VFS. |
| `t3s:table` | A structured data object or schema contract for internal or external data. |
| `t3s:schema` | A formal definition of data structures and validation rules for tables or registries. |
| `t3s:template` | A pre-configured artifact stencil used to increase consistency in production. |
| `t3s:policy` | A control artifact that sets mandatory constraints (Policy-as-Code). |
| `t3s:precedence` | A logic-gate artifact defining the hierarchy of rule resolution. |
| `t3s:utility` | General-purpose tooling, scripts, or helper programs that support development, maintenance, or operational workflows but are not part of the core SDK runtime. |
| `t3s:service` | A logical grouping of related artifacts that fulfill a specific functional outcome. |
| `t3s:record` | An immutable, point-in-time snapshot of an event, audit, or state change. |
| `t3s:proof` | A cryptographic attestation or ZK-proof of compliance/validity. |
| `t3s:protocol` | A specification for external system exchange (e.g., MCP, API contracts). |
| `t3s:steward` | A persistent identity-class URN representing an owner or authority. |
| `t3s:asset` | Static binary data or media (images, PDFs) supporting the documentation. |
| `t3s:gateway` | An interface definition for external API or webhook interactions. |
| `t3s:session` | A transient, stateful instance of an agent interaction within a VFS mount. |
| `t3s:snapshot` | A cryptographically sealed, point-in-time view of a graph or branch. |
| `t3s:module` | Executable artifacts (compiled binaries, shared libraries, or tightly coupled SDK components) residing in the 910_bin directory. Modules represent trusted, versioned execution units of the T3S SDK and related runtime components. |
| `t3s:mirror` | A read-only clone of an external compliance standard or documentation set. |
| `t3s:reference` | A composite object comprising a `t3s:predicate`, `t3s:target`, `t3s:fragment`, and `t3s:description`. It enables granular, cross-plane navigation. |
| `t3s:predicate` | The functional operator (property) within a reference that defines the logic gate behavior. |

## 050.100-001::020 t3s:predicate
A structural primitive that defines the directional logic, semantic weight, and inheritance rules of a t3s:reference. It serves as the functional operator for the T3S Graph, transforming static URN associations into non-discretionary execution constraints within the VFS. `t3s:weight` and `t3s:vector` apply only to t3s:transversal references.

| predicate | definition |
| :--- | :--- |
| `t3s:contains` | **Vertical (Down):** Defines physical/logical inclusion within a branch or root. |
| `t3s:context` | **Vertical (Up):** Child declaration of membership and inheritance from an authoritative anchor. |
| `t3s:dependency` | **Associative:** A functional prerequisite required for mechanical execution, regardless of plane or root residency. |
| `t3s:transversal` | **Multidimensional:** A link that pierces plane isolation to establish cross-functional proximity. |

## 050.100-001::030 Operational Guardrails (VFS Kernel)
Constraints enforced at the protocol/kernel layer:

1. **Conditional Inheritance:** Artifacts inherit metadata from their `t3s:context` by default, but explicit local attributes or `t3s:transversal` requirements take precedence according to the Hierarchy of Precedence.
2. **Deterministic Conflict Resolution:** When transversal and vertical constraints collide, the kernel resolves the state based on `t3s:authority` weights and 010-series Mandates.
3. **Execution Denial:** The SDK inhibits mounting artifacts unless they satisfy the **Validation Stack**: (1) `t3s:integrity` verification, (2) `urn` resolution, (3) `t3s:liability` threshold check, and (4) `t3s:sovereignty` alignment.
4. **Dissonance Handling:** Relationship violations are handled according to the defined `t3s:enforcement` level.

* **`t3s:authority` vs `t3s:weight`:** `t3s:authority` is a node-level precedence used in deterministic conflict resolution between artifacts. `t3s:weight` is an edge-level precedence used only for t3s:transversal links.
* **`t3s:enforcement` scope and composition:** `t3s:enforcement` is attached to artifacts and/or references. When multiple constraints apply, the most severe level (hard-fail > soft-warn > log-only) governs the kernel’s behavior.

*Kernel conflict resolution and execution denial MUST respect the 010-001::130 Hierarchy of Precedence.*

## 050.100-001::040 Functional Attributes
Properties defining the metadata used to steer, categorize, and observe the system state.

| property | definition |
| :--- | :--- |
| **`urn`** | The persistent identifier. Syntax: `urn:t3s:<version>:<type>:<plane>:h:<hash>`. |
| `t3s:namespace` | The Fully Qualified Name (FQN) dot-notation string. |
| `t3s:root` | **Locality:** The top-level filesystem origin forming the first segment of the namespace. |
| `t3s:domain` | **Locality:** The primary functional namespace slug. The second segment of the namespace. |
| `t3s:member` | **Locality:** The normalized terminal slug. Identifies the entity within a namespace. |
| `t3s:index` | **Locality:** The numeric identifier used for sorting and path anchoring. |
| `t3s:anchor` | **Locality:** Identifies the definitional `-000` index file of the local branch or root. |
| `t3s:plane` | **Functional:** The logical layer (control, operation, observation), decoupled from residency. |
| `t3s:vector` | **Traversal:** The dimension of a transversal link (e.g., evolution, compliance, telemetry). |
| `t3s:weight` | The strength or priority of a transversal link (0–100) used for automated conflict resolution within a given t3s:vector. |
| `t3s:parity` | Defines if the transversal link is bidirectional or unidirectional. |
| `t3s:constrains` | **Control:** Declares that a `t3s:policy` or `t3s:precedence` artifact imposes non-discretionary constraints on the target artifact’s execution. |
| `t3s:enforcement` | Defines kernel behavior on failure: **hard-fail**, **soft-warn**, **log-only**. |
| `t3s:status` | Lifecycle state: **draft**, **review**, **active**, **stable**, **immutable**, **deprecated**, **retired**. The immutable state indicates a cryptographically sealed artifact; content changes require a new artifact identity. |
| `t3s:provenance` | Metadata describing the origin, authorship, and derivation context of an artifact, including human and system actors.
| `t3s:criticality` | Operational importance: **tier-1**, **tier-2**, **tier-3**. |
| `t3s:liability` | Risk profile: **low**, **medium**, **high**, **catastrophic**. |
| `t3s:integrity` | The SHA-256 hash of the content below the Stewardship Boundary. |
| `t3s:target` | The authoritative URN or Hash ID of the destination entity or fragment. |
| `t3s:fragment` | The stored Immutable Hash ID (^id) for deep-linking into a specific byte-range or block. |
| `t3s:description` | A narrative attribute providing context. In headers, it summarizes the artifact; in references, it justifies the link.
| `t3s:lineage` | A summary array of archival strings (e.g., `v0.1.0-alpha_h_1a2b3c`) representing the artifact's full evolutionary path. |

## 050.100-001::070 Identity and Namespace Resolution
The **URN** is the *decoupled*, authoritative identifier. It contains NO locality-based descriptors.

**Resolution Precedence:**
1. **Authoritative Lookup:** The system uses the `urn` to resolve the current physical path via the Manifest.
2. **Explicit Namespace:** If `t3s:namespace` is defined, it is used for logical routing and SDK discovery.
3. **Derivative Namespace Logic:** Concatenate `t3s:root`, `t3s:domain`, and intermediate branch slugs.

**Normalization Rule:** All logical namespace segments and the `t3s:member` value MUST replace underscores (`_`) with hyphens (`-`).

**t3s:root:** The `t3s:root` type denotes a top-level node in the graph; the `t3s:root` property stores the slug of that node for namespace derivation.
