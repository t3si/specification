# 040.100-001 System Trusted Keys Registry

## 040.100-001::010 Purpose
This registry defines the **System Trusted Keys**, forming the cryptographic root-of-trust for all steward-class identities.  
It ensures that all high-liability actions, lifecycle transitions, and mandate updates are attributable to a verifiable, sovereign identity.

This registry supports:

- Objective C — Digital Sovereignty  
- Objective G — Provenance Integrity  
- Objective H — Lineage Tracking  
- Objective K — Deterministic Enforcement  
- Objective N — Recursive Self-Correction  

All steward-class identities MUST be defined here before they may perform any `tier-1` action.

### Steward Identity Requirements
1. The steward URN MUST appear in this registry before performing any `tier-1` action.  
2. The steward’s public key MUST match the fingerprint listed above.  
3. The steward’s `status` MUST be `active` for authorization to succeed.  
4. All steward actions MUST update `t3s:provenance` and `t3s:lineage` in affected artifacts.

## 040.100-001::030 Authorization Mandate
This section establishes the technical constraints for identity verification and state-change authority.

1. **Sovereign Command**  
   All artifacts designated as `criticality: tier-1` MUST be signed by a key whose fingerprint matches an `active` entry in this registry.

2. **Zero-Knowledge Addressing**  
   The URN  
   `urn:t3s:v0.2.1:<type>:<plane>:h:[hash]`
   is the immutable identifier for this registry, decoupling identity from physical pathing.

3. **Authorization Logic**  
   A signature assertion is only valid if:
   - the steward URN exists in this registry  
   - the signature matches the registered fingerprint  
   - the identity `status` is explicitly `active`  
   - the steward is authorized for the target artifact’s `t3s:criticality` level  

4. **Lifecycle Integration**  
   Steward identities may be transitioned to `deprecated` or `retired` states, but such transitions MUST:
   - be signed by another active steward  
   - update the registry’s `t3s:lineage`  
   - trigger a `040-000` integrity audit  

## 040.100-001::040 Revocation & Rotation Protocol
To maintain sovereign integrity:

1. **Revocation**  
   If a steward key is compromised, the identity MUST be marked `retired` and all affected artifacts revalidated.

2. **Rotation**  
   A steward may rotate keys by:
   - adding a new key entry  
   - marking the old key as `deprecated`  
   - updating provenance and lineage  

3. **Forensic Continuity**  
   All revocation and rotation events MUST be recorded as immutable `t3s:record` artifacts.
