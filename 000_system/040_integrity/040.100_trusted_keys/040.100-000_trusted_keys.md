# 040.100-001 Trusted Keys Anchor

## 040.100-001::010 Purpose
This anchor establishes the **Trusted Keys Root** for the T3S Control Plane.  
It defines the persistence, governance, and verification rules for all steward, service, and environment identities.

The Trusted Keys Anchor supports:

- Objective C — Digital Sovereignty  
- Objective G — Provenance Integrity  
- Objective H — Lineage Tracking  
- Objective K — Deterministic Enforcement  
- Objective N — Recursive Self-Correction  

All subordinate registries in the 040.100 series inherit their authority from this anchor.

## 040.100-001::020 Identity Persistence Mandate
All identities within this branch utilize **Functional Hashes** (`h:<hash>`) to ensure URN persistence and cryptographic decoupling.

1. **Separation of Concerns**  
   The URN defines *identity* (who/what), while subordinate registries (e.g., 040.100‑001) define the **validity state**, **public key**, and **cryptographic proof**.

2. **Status-Gated Verification**  
   Possession of a valid key is insufficient for authorization.  
   The registry MUST explicitly designate an identity as `active` to satisfy the enforcement requirements of **040‑000 Access and Integrity**.

3. **Path Agnosticism**  
   Identity hashes are derived from logical hierarchy segments and remain static regardless of physical file-system migrations, repository restructuring, or VFS remounts.

4. **Provenance Integration**  
   Any change to an identity’s validity state MUST update `t3s:provenance` and `t3s:lineage` in the corresponding registry entry.

## 040.100-001::030 Subordinate Registries
The Trusted Keys Anchor delegates authority to the following registries:

- **System Keys — `040.100-001`**  
  Genesis Root; Tier‑1 Steward Authority; canonical source for human and steward-class identities.

- **Environment Keys — `040.100-002`**  
  Deploy targets, infrastructure nodes, and environment-scoped service identities.

Each subordinate registry MUST:

- maintain deterministic `t3s:integrity` seals  
- enforce lifecycle transitions (`draft` → `active` → `retired`)  
- record provenance for all key additions, rotations, and revocations  
- comply with the enforcement semantics defined in **010‑001** and **020‑000**

## 040.100-001::040 Sovereign Boundary Rules
To preserve the sovereignty of the T3S graph:

1. No identity may impersonate another.  
2. No environment key may escalate to steward authority.  
3. No service identity may modify `tier‑1` artifacts.  
4. All identities MUST be resolvable through the 070‑series Manifest.  
5. All identity actions MUST be cryptographically attributable.

These rules ensure that the Trusted Keys Anchor remains the immutable root of identity governance.
