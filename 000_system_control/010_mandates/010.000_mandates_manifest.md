# 010 Policy and Mandates Manifest

## 1.0 Purpose
This manifest defines the foundational mandates that govern the T3S framework. In alignment with ITIL 4 **Governance**, it establishes the "Rules of Engagement" for the system, ensuring that all architectural and operational decisions remain compliant with the initiative's core ethics and legal obligations.

## 2.0 The Five Immutable Mandates
All artifacts within this repository must satisfy these mandates to be promoted to `Active` status:

### M1: Deterministic Sovereignty
The system must be traversable and auditable without reliance on proprietary, third-party black-box logic. Every "leaf" must be traceable to an "anchor."
* **Alignment:** Objective A (Deterministic Mapping).

### M2: Ontological Integrity (The Markov Blanket)
No change shall be permitted that compromises the structural boundary of the system. External dependencies must be explicitly declared and isolated.
* **Alignment:** Objective C (Digital Sovereignty).

### M3: Verifiable Provenance
Every contribution must be linked to a verified human or authorized entity through cryptographic signatures. Anonymous "ghost" logic is strictly forbidden.
* **Alignment:** Objective E (Verifiable Insurability).

### M4: Semantic Stewardship
Data and logic must be self-describing via JSON-LD or YAML manifests. Ambiguity is treated as a high-risk system defect.
* **Alignment:** Objective B (Semantic Stewardship).

### M5: Privacy-First/Local-First
Processing of sensitive governance logic should occur within the local sovereign layer. Data leakage to non-sovereign LLMs or cloud providers must be mitigated by the **060 Pipelines**.
* **Alignment:** Objective D (Sovereign Infrastructure).

## 3.0 Compliance Framework
In accordance with ITIL **Service Configuration Management**, these mandates serve as the "High-Level Control" requirements. 
* **Verification:** Compliance is audited via the `060.000_governance_pipelines`.
* **Escalation:** Violations of these mandates must be recorded as an **Emergency ADR** in the `140_adr` block.

## 4.0 The Principle of Semantic Parity
To maintain the integrity of the **System Graph**, the repository must exhibit absolute parity between its physical structure and its semantic definitions:
1. **Structural Alignment:** Every directory indexed in the 070.000 Root Manifest must exist in the physical file system.
2. **Identifier Consistency:** URNs and IDs must be unique and consistent across all planes (Control, Observational, and Operational).
3. **Forensic Traceability:** Any modification to the system's structure must be preceded by an update to the corresponding manifest to ensure the "Map" guides the "Change."

## 5.0 The Dual-Stack Documentation Mandate
To ensure the system remains both Sovereign (human-governed) and Deterministic (machine-verified), all System Blocks must adhere to the Dual-Stack standard:

1. **Human-Readable Primacy (The Anchor):** The primary entry point for every block (the .000 index) MUST be a Markdown (.md) file. This file serves as the "Source of Intent," containing the mandates, strategy, and ITIL alignment necessary for a human Steward to exercise oversight.
2. **Machine-Readable Parity (The Manifest):** The functional mapping of the block (paths, URNs, and metadata) MUST be codified in the 070.000 Root Manifest (JSON-LD). This serves as the "Source of Logic" for automated pipelines and agentic interaction.
3. **Semantic Sync:** In the event of a conflict between the Markdown intent and the JSON logic, the Markdown mandate takes precedence until the discrepancy is resolved via a signed ADR (140).
