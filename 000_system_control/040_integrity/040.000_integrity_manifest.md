# 040 Access and Integrity Manifest

## 1.0 Purpose
This manifest defines the cryptographic and procedural requirements for ensuring the integrity of the T3S framework. It establishes the "Technical Guardrails" for identity verification, commit signing, and non-repudiation, aligning the repository with ITIL 4 Information Security Management standards.

## 2.0 Cryptographic Root of Trust
To ensure the **Markov Blanket** remains closed to unauthorized or anonymous influence, the following standards are mandated:

### 2.1 Commit Signing (Non-Repudiation)
* **Standard:** All commits MUST be signed using a GPG or SSH key.
* **Verification:** The public keys of authorized System Stewards must be indexed in the `040.100_trusted_keys/` leaf directory.
* **Enforcement:** The `060.000_governance_pipelines` shall reject any commit lacking a valid cryptographic signature.

### 2.2 Identity Mapping
* **Identifier:** Each contributor must be associated with a unique URN (e.g., `urn:t3s:steward:[USERNAME]`).
* **Parity:** The email address associated with the cryptographic key must match the `Signed-off-by` header in the Git DCO (Developer Certificate of Origin).

### 2.3 Secret Segregation Mandate
* **Requirement:** No private keys, passwords, or raw cryptographic entropy shall be stored within the T3S repository.
* **Requirement:** Private keys must be managed in a local-first, "Sovereign" vault (as per 900-block requirements) that exists outside the version-control history.
* **Requirement:** The repository only stores the *proof* of identity (the signature), never the *source* of identity (the key).

### 2.4 Service-Class Identity Exceptions (Resilience)
While Tier-1 governance changes (000-100 blocks) strictly require Steward-class GPG signatures, the system permits **Service-Class Identities** for automated operational logging (500 block) provided that:
1. **Scoped Authority:** The identity is restricted to "Append-Only" permissions within specific operational directories.
2. **Cryptographic Traceability:** The service key is indexed in `040.200_service_keys/` and linked to a parent Steward-class anchor.
3. **Non-Repudiation:** Automated logs must still carry a valid signature to prevent "Ghost Telemetry" within the Markov Blanket.

## 3.0 Access Control Levels
In accordance with the principle of **Least Privilege**, the following access tiers are defined:

| Tier | Role | Capabilities |
| :--- | :--- | :--- |
| **Tier-1** | System Steward | Merge authority; GPG Root holder; Authority to promote to `Active`. |
| **Tier-2** | Contributor | Read/Write access to `Draft` and `Genesis` artifacts via PR. |
| **Tier-3** | Auditor | Read-only access to forensic evidence and manifests. |

## 4.0 Integrity Assertion (ITIL 4 SVS)
The System Steward shall perform a monthly **Integrity Audit** to ensure:
1. **Chain of Custody:** No artifact has been modified without an associated and signed **140_adr**.
2. **Key Rotation:** Periodic review of active public keys in the `040_integrity` block.
3. **Breach Protocol:** In the event of a compromised key, the steward must issue a `Retired` status to the affected identity and re-verify all commits since the last known "Good State."
