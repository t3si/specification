# 060 Governance Pipelines and Gates

## 1.0 Purpose
This standard defines the automated "Checkpoints" required to maintain the system's state. It translates the **120.000 Standard of Care** into executable validation logic.

## 2.0 The "Genesis Gate" (CI Workflow)
Every Pull Request or local merge must trigger the **Genesis Gate**, consisting of:
1.  **Linter (Path & Index):** Verify that all new files lead with a 3-digit prefix and follow the `XXX.XXX_slug` format.
2.  **Schema Validator:** Validate all `.json`, `.jsonld`, and `.yaml` files against the **130.000** schema requirements.
3.  **Provenance Check:** Ensure the `Signed-off-by` DCO header is present and the GPG signature is valid.

## 3.0 Verification of Assertions
The pipeline must return a `SUCCESS` signal only if the following ITIL-aligned conditions are met:
* **Assertion A0 (Structure):** 100% of files are indexed.
* **Assertion A1 (Provenance):** 100% of commits are signed and attributed.
* **Assertion A2 (Semantic):** The `070.000_root_manifest.jsonld` correctly resolves all current repository paths.

## 4.0 Failure Protocols
Any artifact failing a Gate is automatically assigned the status `Invalid`. It may not be merged into the `main` branch or promoted to `Active` status in the `020.000_t3s_manifest.json`.
