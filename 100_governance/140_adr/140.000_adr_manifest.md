# 140 ADR Manifest Standard

## 1.0 Purpose
This standard defines the process for capturing and persisting significant architectural decisions. ADRs provide the "Ontological Context" for the repository, preventing regressive changes and ensuring that all system evolutions are anchored in documented reasoning.

## 2.0 The Decision Lifecycle
All ADRs must progress through the following statuses:
* **Proposed:** The decision is under review by the System Steward or T3SI partners.
* **Accepted:** The decision is ratified and the repository has been aligned to reflect it.
* **Superseded:** A newer ADR has rendered this decision obsolete (must link to the new ADR).
* **Deprecated:** The decision is no longer normative but remains for forensic history.

## 3.0 Structural Mandates
Every ADR leaf (e.g., `140.001_...`) must adhere to the **130.000 Manifest Schema** and include:
1.  **Context:** The technical or regulatory pressure that necessitated the decision.
2.  **Decision:** The specific action taken (The "Sovereign Command").
3.  **Consequences:** The impact on the system (e.g., increased path complexity, improved stability).
4.  **Verification:** Which **Standard of Care (120)** assertion this decision satisfies.
